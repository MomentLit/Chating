# AI Usage Guide

## Principles

AI may be used, but code generated by AI cannot be submitted if the developer cannot explain it.
Project Docs and existing code structure override general AI recommendations.
Issue and PR responsibility belongs to the human developer, not the AI.
If an implementation changes API behavior, update `API_SPEC.yaml` in the same PR.

## How To Use AI

- Use AI to draft Issues, implementation plans, tests, self-reviews, and API specification updates.
- Do not ask AI to write code immediately.
- Ask AI for an implementation plan first.
- Do not blindly paste AI-generated code.
- Do not submit code you cannot explain.
- Ask AI to identify unclear requirements instead of guessing.

## Issue Drafting

Use AI to turn feature requests into clear Issues with acceptance criteria, exception cases, API impact, and open questions.

## Code Self-Review

Use AI after implementation to find architecture drift, naming inconsistency, missing API spec updates, missing tests, and unnecessary changes.

## API_SPEC.yaml Updates

Use AI to update OpenAPI content from controller, request DTO, response DTO, and service behavior changes. Do not invent behavior that is not visible from code or explicitly approved.

## Implementation Plan Prompt

```text
Create an implementation plan based on the project documentation and existing structure.

Reference documents:
- docs/service-overview.md
- docs/service-policy.md
- docs/common/architecture-guide.md
- docs/common/coding-convention.md
- docs/common/ai-usage-guide.md
- API_SPEC.yaml

Issue:
[Paste Issue content]

Rules:
- Do not change the existing architecture.
- Modify only necessary files.
- Separate Controller, Service, Repository, and DTO responsibilities.
- Do not write code yet.
- If requirements are unclear, list questions instead of guessing.
- If API behavior will change, include the required `API_SPEC.yaml` updates in the plan.

Output:
1. Files to change
2. Reason for each change
3. Implementation steps
4. Required exception handling
5. Test or verification items
6. Required API_SPEC.yaml updates
7. Unclear requirements
```

## AI Self-Review Prompt

```text
Review the following code against our project rules.

Reference documents:
- docs/service-overview.md
- docs/service-policy.md
- docs/common/architecture-guide.md
- docs/common/coding-convention.md
- API_SPEC.yaml

Review criteria:
1. Architecture violations
2. Exception handling issues
3. DTO usage issues
4. Naming issues
5. Missing tests or verification
6. Unnecessary changes
7. Inconsistency with existing code style
8. API behavior changes not reflected in API_SPEC.yaml

Code:
[Paste changed code or diff]
```

## API Spec Update Prompt

```text
Update API_SPEC.yaml based on the following API changes.

Existing API_SPEC.yaml:
[Paste current API_SPEC.yaml]

Code changes or intended API behavior:
[Describe API changes or paste controller/request/response code]

Rules:
- Use OpenAPI 3.x format.
- Do not invent behavior that is not visible from code or explicitly specified.
- Use `Needs confirmation` for uncertain items.
- Update paths, parameters, requestBody, responses, schemas, and error responses as needed.
- Keep the API spec consistent with the implementation.

Output:
1. Updated API_SPEC.yaml
2. Summary of API changes
3. Items that still need confirmation
```
